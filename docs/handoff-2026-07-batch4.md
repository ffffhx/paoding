## D1：多菜同做时间线 v1（差异化功能，竞品只有 SideChef 对自家人工内容做过）

**场景**：做一桌菜（如周计划某天的 2–4 道菜），自动生成合并时间线：哪道菜先动手、哪些等待时间可以穿插做别的菜，让所有菜尽量同时上桌。

**要做**：
1. `mergeCookTimeline(recipes)` 纯函数（前端可测试区）：
   - 每步时长：复用现有计时器的时长识别；无时长的步骤给默认值（如 3 分钟）并标 `estimated`。
   - 被动步骤判定：步骤文本命中 炖/焖/腌/烤/蒸/煮/静置/发酵/晾/浸泡 等关键词 → 被动等待（此期间可做其他菜的主动步骤）。
   - 输出：按时间偏移排序的动作列表 [{offsetMin, recipeTitle, stepIndex, text, passive}]，目标是所有菜同时完成（从完成时刻倒排）。
   - 算法从简：主动步骤不并行（一个人只有一双手），被动等待期间可插入其他菜的主动步骤。
2. UI：本周计划某天（≥2 道菜时）出「同做时间线」按钮 + 菜谱多选入口；展示为顺序列表（"第 X 分钟 · 菜名 · 步骤摘要"，被动步骤标 ⏳），不做甘特图。
3. 测试：两道菜含被动步骤的穿插排布、无时长兜底、单菜退化为原顺序。

## D2：菜谱编辑体验增强（竞品差评集中点）

**现状**：可编辑标题/用量/步骤/讲解文本，但不能调整结构。

**要做**：
1. 编辑模式下：步骤与食材支持**上移/下移**（按钮式，不做拖拽，移动端可靠）、**新增**一条、**删除**一条（删除要确认）。
2. 菜谱标签编辑：详情页可增删标签（供 C2 的筛选使用）。
3. 数组移动/增删纯函数（moveItem/insertItem/removeItem）放可测试区，配测试；保存沿用现有"保存即同步"。
4. 注意步骤重排后 `source_time`、图片、why 跟随步骤对象一起移动（整个对象移动就不会错位，测试覆盖这一点）。

## D3：多用户 household 隔离 v1（结构性缺口，Mealie 对标）

**现状**：全局单份 `paoding-userdata.json`，两个人共用后端会互相覆盖收藏/购物清单。

**要做**：
1. `PAODING_API_TOKENS` 环境变量：`alice:token1,bob:token2` 格式定义多用户。向后兼容：只设旧 `PAODING_API_TOKEN` 时等价于单个默认用户，行为与现在完全一致（含 userdata 文件名不变）。
2. 鉴权时解析出用户名（逐个 token 用 `crypto.timingSafeEqual` 比较，保持常量时间语义）；请求上下文带用户。
3. userdata 按用户分文件：默认用户沿用 `paoding-userdata.json`，其他用户 `paoding-userdata-<user>.json`（user 名做 basename 白名单校验，防路径注入）。B4 的 rev 机制每用户独立。
4. **菜谱库仍全局共享**（家庭场景合理，不做菜谱隔离）；收藏/笔记/评分/购物清单/排菜/最近任务视图按用户隔离。
5. 限流维度从 IP 升级为 IP+用户。
6. 测试：双 token 各自 PUT/GET userdata 互不串扰；旧单 token 配置行为不变；非法 user 名被拒。
7. README「多用户」说明 + `.env.example`。
