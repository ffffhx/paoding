## C1：膳食计划营养汇总

**现状**：P3 已给菜谱落库 `nutrition.per_serving`（calories_kcal/protein_g/fat_g/carbs_g/sodium_mg，estimated 标记）；膳食计划是周日历排菜（前端 mealPlan），与营养完全脱节。

**要做**：
1. 周计划视图每天显示当日已排菜谱的营养合计（只汇总**有** nutrition 缓存的菜，缺的标"N 道菜未估算"，不臆造），按用户设置的份量因子换算。
2. 周视图给一行整周日均值。样式轻量（小字附注即可），暗色模式适配。
3. 汇总纯函数（输入：菜谱数组+份量因子 → 输出：合计与缺失计数）放可测试区，配前端测试。

## C2：菜谱库搜索/筛选/排序增强

**现状**：列表有搜索/标签筛选。竞品用户抱怨最多的是"不能按食材筛、排序选项少、无子分类"。

**要做**：
1. 按食材筛：输入食材名（支持多个，逗号分隔），筛出食材表中含全部关键词的菜谱（子串匹配即可）。
2. 排序选项：最近添加（默认）/ 评分 / 总时长（菜谱 JSON 若有 total_time 或步骤时长可加总；没有就标"未知"排最后）/ 名称。
3. 快捷筛选 chips：已收藏 / 做过 / 未做过 / 有营养信息。
4. 筛选+排序纯函数化并配测试；UI 控件放列表页顶部，保持现有风格，移动端不挤。

## C3：导入互通（只出→进出双向）

**现状**：只有导出（Cooklang `.cook`、schema.org JSON-LD、自家备份 JSON）。竞品标配"从别家迁移进来"；开源生态（Mealie/Tandoor）以 schema.org 为通用交换格式。

**要做**：
1. `POST /api/import-recipe`：接受单个 schema.org/Recipe JSON-LD（兼容 `@graph` 包裹、HowToStep/HowToSection 步骤、ISO 8601 时长），映射成庖丁菜谱 JSON 落库。没有的字段（why 讲解、source_time、置信度）留空并标 `imported: true`，**不臆造 why**；菜谱详情页对 imported 菜显示"外部导入，无原理讲解"并提供一个"AI 补讲解"按钮（复用现有 explain 逻辑按钮式触发，走现有限流）。
2. 前端「导入」入口：粘贴 JSON-LD 文本或选 `.json` 文件。
3. 解析/映射纯函数放 `src/`（如 `src/importRecipe.mjs`），覆盖测试：标准 Recipe、@graph 包裹、HowToSection 嵌套、脏数据（缺字段/类型错误）不崩。
4. Cooklang 导入暂不做（语法解析成本高、收益低），README 里说明支持范围。

## C4：跨视频技法库 v1（Roadmap 差异化项）

**目标**：把不同菜谱里反复出现的技法（焯水、滑油、炝锅、收汁、上浆、煸炒…）沉淀成可浏览的「技法库」，每个技法聚合：它在哪些菜里出现过、各菜里对应步骤的"为什么"讲解、跳转入口。

**要做（v1 从简，纯离线聚合，不加解析管线开销）**：
1. `src/techniques.mjs`：维护一份中餐常见技法词表（30–50 个，含别名，如 焯水/飞水/汆water），提供 `extractTechniques(recipe)` 纯函数——扫描步骤文本与 why 文本，命中词表即归档 {technique, recipeId, stepIndex}。
2. `GET /api/techniques`：服务端对全部菜谱在线聚合（菜谱量级小，实时算即可，不引入索引文件），返回技法→出现列表。
3. 前端新增「技法」页（入口放底部导航或首页），列出命中的技法及次数；点开看聚合的各菜步骤讲解摘录 + 跳到对应菜谱步骤。
4. 词表与提取纯函数配测试。README 功能表 + Roadmap 勾选更新。
