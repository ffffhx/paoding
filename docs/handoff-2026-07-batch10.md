## J1：食材同音字纠错（三层防御）

**第一层：结构化 prompt 纠错**（src/chef.mjs SYSTEM）
新增规则：「【语音转写】来自语音识别，含同音/近音错别字，尤其是香料和食材名。整理时必须按烹饪常识纠正为正确写法（例：白纸→白芷、肉豆扣/肉豆寇→肉豆蔻、草扣→草蔻、山奈→山柰）。纠正过的食材在 note 里标注：转写作「原词」，已按烹饪常识纠正。拿不准是不是错别字时保留原词，不要乱改。」

**第二层：领域词表兜底**（新增 src/ingredientFix.mjs）
- 常见中餐香料/食材的 ASR 高频错写映射表（数据驱动，30+ 条起步）：白纸/白直→白芷、肉豆扣/肉豆寇/肉蔻→肉豆蔻、草寇/草扣→草蔻、山奈→山柰、必拨/毕拨→荜拨、陈皮/成皮、香叶、丁香/钉香、砂仁/沙仁、桂皮/贵皮、八角/巴角、花椒/花教、孜然/资然、豆蔻、良姜/凉姜、罗汉果、甘草/干草、当归、党参、黄芪/黄奇 等（按拼音同音原则自行补全，宁少勿错——**只收无歧义的**：像「干草→甘草」这种在做菜语境几乎必然是香料的才收；有多义可能的不收）。
- `fixIngredientName(name)` 纯函数：命中映射返回 {name: 正确名, corrected: true, original: 原词}；`structureRecipe` 之后对 ingredients[].name 和 tools 无关（工具不用）跑一遍，纠正时往 note 追加标注（与第一层同格式，去重）。步骤文本里的同款错词也替换（只替换映射表命中的完整词）。
- 测试：映射命中/不命中/note 追加/步骤替换。

**第三层：AI 助手端点防御**（app/server.mjs，兜住已入库的旧菜谱）
- `/api/substitute`（以及 ask/term）prompt 追加：「食材名可能包含语音识别的同音错别字（如 白纸 实为 白芷、肉豆扣 实为 肉豆蔻）。先判断名字是否为误写：若是，回答开头先指出正确名称，再按正确食材给替代建议；若名字本身不是食材也无法推断，直说无法识别，不要硬编。」
- 测试：LLM stub 断言 prompt 含防御指令。

## J2：生活化定量（身体参照量词）

**结构化 prompt 增强**（src/chef.mjs）：
1. 新增规则：「口播里的生活化定量描述是最重要的用量信息，必须原样保留进 amount：拇指长、小拇指粗、指甲盖大（视频若说明是哪个手指要保留）、一元硬币大小、鸡蛋大小、乒乓球大小、巴掌大 等。例：转写说『姜切拇指长一段』→ amount 写 拇指长一段，不得简化成 一段。」
2. 新增规则：「食材只给了 一片/一段/一块/一把 这类模糊量、且转写里没有大小参照时：amount 保留原文，并在 note 里给一个常识参考大小，须以「参考：」开头并标注推测性质（例：姜一片 → note: 参考：约硬币大、2毫米厚（常识推测））。转写里已有大小参照时绝不覆盖。」
3. steps 里的到位判断/操作描述同样保留身体参照描述，不得抽象化。
4. 测试：LLM stub 场景——转写含「拇指长」断言 amount 保留；模糊量断言 note 有「参考：」。

## J3：替代品结果缓存（用户反馈：本地模型思考一次太久，不想重复思考）

**照 P3 营养缓存的既有模式**（缓存直返 → 无则 LLM → 写回菜谱 JSON → 编辑失效）：

1. `/api/substitute` 改造：菜谱 JSON 增加 `substitutes: { "<食材名>": { answer, created_at } }` 缓存。请求时：命中缓存直接返回（响应里带 `cached: true`）；未命中才调 LLM，成功后写回菜谱 JSON 再返回。
2. 失效策略：编辑菜谱时若食材列表变化（食材名集合变了），清掉已不存在食材的缓存条目；J1 的纠错改名同样视为变化。支持强制刷新：请求带 `force: true` 跳过缓存重新生成并覆盖。
3. 前端：
   - 弹窗打开时命中缓存 → **秒出结果**，底部小字「已缓存 · 重新生成」可点（带 force 重打）；
   - 未命中 → 现有「思考中…」流程，出结果后自动落缓存；
   - 新文案走 t()（zh+en）。
4. `/api/ask` 自由问答不缓存（每次问题不同）；`/api/term` 术语解释同样按本节模式加缓存（key 是术语本身，收益一样大）。
5. 测试：LLM stub——首次调用走 LLM 并落盘；二次调用不打 LLM 直返（stub 计数断言）；force 重新生成；编辑食材后对应缓存失效。
