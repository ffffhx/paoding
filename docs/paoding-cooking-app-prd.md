# 庖丁 · 做菜视频「解剖」App — 需求文档 & 技术方案

> 状态：v0.2 — 需求已确认，MVP 引擎已落地（见第 9 节）
> 定位：个人项目，MVP 优先，先做出来自己用；架构预留后续商业化空间
> 一句话：**把任意做菜视频，自动拆成「可逐步打卡的步骤清单」，并对每一步讲清楚"为什么这么做、不这么做会怎样"。**

---

## 0. 为什么做这个（市场空白）

把这个想法拆成三块能力，各自的市面现状：

| 能力 | 代表产品 | 现状 |
| --- | --- | --- |
| ① 视频 → 图文菜谱步骤 | Pluck、Video2Recipe、Cooking Guru、CookNotebook、食神 | **已成熟**，但产出是"扁平菜谱"，只告诉你做什么，不讲为什么 |
| ② 分步"跟做"模式 | Mise、Pestle、Paprika、Fond、Crouton | **已成熟**，一次显示一步 / 屏幕常亮 / 语音翻步 / 一键计时 |
| ③ 讲透"为什么 / 不这么做会怎样" | America's Test Kitchen、ChefSteps | **稀缺**，且是编辑部人工写的，不吃视频、不支持用户上传 |

**结论：没有任何一款产品把①②③缝在一起。** 现有"视频转菜谱"止步于①；有深度原理的又不吃视频。
**庖丁的差异点 = 自动解析（①）× 深度原理教学（③），再套上顺手的跟做壳（②）。**

---

## 1. 目标用户 & 核心场景

- **主用户**：想认真学做菜、不满足于"照抄步骤"、想搞懂原理的家庭做饭者（含你自己）。
- **典型场景**：
  1. 刷到一个抖音/B站做菜视频 → 分享/粘贴链接进庖丁 → 秒得结构化步骤。
  2. 开火做饭时，手机进"跟做模式"，做完一步点"下一步"，手脏时语音说"下一步"。
  3. 对某步有疑问（"为什么油要冒烟？"）→ 点开该步的「为什么」→ 看原理 + "不这么做会怎样"。
  4. 把讲得好的步骤 / 整道菜 **收藏**，形成自己的"厨艺知识库"。

## 2. 核心功能需求

> 标注：🟢 = MVP 必做；🟡 = V1；⚪ = V2+

### 2.1 视频导入
- 🟢 粘贴链接：抖音、小红书、B站
- 🟢 粘贴链接：YouTube、Instagram、TikTok
- 🟢 本地上传视频文件
- 🟡 系统"分享到庖丁"（从抖音/B站直接分享，不用手动复制链接）
- ⚪ 粘贴纯图文菜谱链接 / 截图 OCR 也能解析

### 2.2 AI 解析 → 结构化菜谱（核心）
- 🟢 输出：菜名、份量、总耗时、食材清单（名称+用量）、**有序步骤列表**
- 🟢 每一步包含：步骤标题、操作描述、关键参数（火候/油温/时间/状态判断，如"油七成热""煎至两面金黄"）
- 🟡 步骤自动关联视频时间戳（点步骤跳到视频对应片段）
- 🟡 食材清单可勾选、可一键生成购物清单
- ⚪ 自动识别菜系 / 难度 / 标签

### 2.3 每一步的「为什么」深度讲解（核心差异点）
- 🟢 每步生成三段式讲解：
  - **为什么这么做**（原理：如美拉德反应、锁水、断生、去腥）
  - **不这么做会怎样**（后果：粘锅、腥、老、不入味）
  - **判断标准 / 小技巧**（怎么知道到位了）
- 🟡 关键术语可点开（"美拉德反应""焯水""滑油"）看微科普
- 🟡 讲解可"追问"：对某步继续问 AI（"没有料酒能用什么替代？"）
- ⚪ 挂烹饪原理知识库（RAG）降低幻觉、提升权威性

### 2.4 跟做模式（Cook Mode）
- 🟢 一次一步、大字显示、**屏幕常亮**
- 🟢 "下一步 / 上一步" 按钮，进度条
- 🟡 语音控制翻步（"下一步""上一步"）——手脏刚需
- 🟡 步骤内一键起计时器（识别"煮8分钟"→点一下开始倒计时）
- ⚪ 语音朗读当前步骤 + 原理

### 2.5 收藏 & 个人库
- 🟢 收藏整道菜
- 🟢 **收藏单个步骤**（你明确提到的需求）——存到"技巧收藏夹"
- 🟡 收藏夹分组 / 搜索 / 标签
- 🟡 解析历史记录

### 2.6 账号 & 同步
- 🟢 MVP 可先本地存储、无账号（最快跑通）
- 🟡 登录 + 云端同步（微信/Apple/手机号）

## 3. 我补充建议的功能（供你挑）

以下不是你原始需求，是我认为能显著提升价值、建议纳入的：

1. **"食材替代"助手** 🟡：没有某个调料时，AI 给可替代方案 + 风味差异说明。承接你"为什么用这个调料"的思路，很自然。
2. **份量换算** 🟡：原视频 2 人份，我要做 4 人份，一键换算所有用量。
3. **"新手雷区"标记** 🟡：AI 把最容易翻车的步骤高亮（如"这一步火太大会糊"），比平铺讲解更抓重点。
4. **解析质量兜底** 🟢：视频信息不足时，明确标出"这步原视频没讲清，以下为 AI 推测"，避免一本正经胡说（诚实比装懂重要）。
5. **多视频合并 / 对比** ⚪：同一道菜看 3 个 up 主，庖丁对比不同做法差异。（护城河功能，远期）
6. **离线可用** 🟡：解析过的菜谱缓存本地，厨房网差也能跟做。

> 建议 MVP 至少纳入 **#4（质量兜底）**，因为个人自用时"AI 瞎编原理"是最伤体验的点。

## 4. 关键数据结构（菜谱 Schema）

整个产品的核心是这套结构，AI 解析的输出、前端渲染、收藏都围绕它：

```jsonc
{
  "recipe_id": "uuid",
  "title": "红烧肉",
  "source": { "type": "douyin|bilibili|youtube|local", "url": "...", "duration_sec": 312 },
  "servings": 2,
  "total_time_min": 60,
  "difficulty": "medium",
  "tags": ["家常菜", "炖菜"],
  "ingredients": [
    { "name": "五花肉", "amount": "500g", "note": "带皮" },
    { "name": "冰糖", "amount": "30g" }
  ],
  "steps": [
    {
      "index": 1,
      "title": "焯水去腥",
      "action": "五花肉冷水下锅，加姜片、料酒，大火煮开撇去浮沫",
      "params": { "heat": "大火", "time": "水开后3分钟", "cue": "浮沫撇净、肉变色" },
      "video_ts": { "start": 12, "end": 48 },   // 关联视频时间戳
      "why": {
        "reason": "冷水下锅让血水和杂质随升温慢慢析出，形成浮沫被撇走",
        "if_not": "热水下锅会瞬间锁住表面，血沫封在肉里，成品发腥、汤浑",
        "cue": "撇到几乎没有灰褐色浮沫即可"
      },
      "risk_level": "low",       // 新手雷区标记
      "confidence": "high"       // AI 对该步解析的置信度；low 时前端提示"可能不准"
    }
  ]
}
```

> 设计要点：`why` 三段式、`video_ts` 支持点步跳转、`confidence` 支撑"诚实兜底"、`params.cue` 是"状态判断"这类做菜真正关键但菜谱常漏写的信息。

## 5. 技术方案

### 5.1 整体架构

```
┌─────────────┐   链接/文件   ┌────────────────────────────────────┐
│  App 双端    │ ───────────▶ │  后端服务 (取流 + 编排 + 缓存)         │
│ (RN/Flutter)│ ◀─────────── │                                    │
└─────────────┘   菜谱 JSON   │  ①视频获取 → ②解析 → ③讲解生成       │
                              └───────┬───────────┬────────────────┘
                                      ▼           ▼
                              [视频源解析]   [AI 模型层]
                              yt-dlp/各平台   多模态/文本 LLM
                                      │           │
                                      ▼           ▼
                              [对象存储]     [菜谱DB + 缓存]
```

### 5.2 前端（双端 App）
- **推荐 React Native + Expo**：一套代码双端，迭代快，生态与本仓库（TS/JS 为主）契合；跟做模式的"屏幕常亮"（`expo-keep-awake`）、语音（`expo-speech` / 原生语音识别）、计时器都有现成能力。
- 备选 Flutter：性能与动画更好，但你现有栈偏 JS，学习成本高些。
- MVP 阶段可先只调一端（如 iOS）验证体验，再补另一端。

### 5.3 后端
- 一个轻量服务（**Node/TypeScript + Fastify** 或 **Python + FastAPI**），职责：接收链接/文件 → 取流 → 编排 AI → 落库 → 返回结构化菜谱。
- 解析是耗时任务（几十秒~几分钟），走**异步任务 + 轮询/推送**：提交后返回 task_id，App 轮询进度（"下载中→解析中→生成讲解中→完成"）。

### 5.4 视频获取（三种源）
| 源 | 方案 | 风险 |
| --- | --- | --- |
| YouTube / TikTok / Instagram | `yt-dlp`（成熟） | 低 |
| B站 | `yt-dlp` 支持较好 | 低 |
| 抖音 / 小红书 | 需专门解析（去水印接口/无水印直链），稳定性一般，需维护 | **中**：平台改版易失效，有版权灰区 |
| 本地上传 | 直传对象存储 | 低，最合规 |

> **建议**：MVP 先把**本地上传 + YouTube/B站**跑通（最稳），抖音/小红书作为快速跟进项，并做好"解析失败优雅降级"（提示用户改用录屏上传）。

### 5.5 AI 解析：两套方案对比（你要的对比）

**方案 A：多模态大模型直接吃视频**

- 做法：把视频（或抽样帧+音轨）直接喂给能理解视频的模型，一次调用出结构化菜谱。
- 可选模型：Gemini（原生支持视频）、豆包·视频理解、通义千问-VL / Qwen-VL-Max。
- ✅ 工程量最小，出效果最快，能捕捉"连续动作"（如翻炒手法）。
- ❌ 长视频 token 贵；"为什么"的准确度依赖模型烹饪常识；视频要上传到模型方（隐私/合规）。

**方案 B：自建管线（抽帧 + ASR + OCR → 文本 LLM 整合）**

- 做法：`ffmpeg` 场景切分抽关键帧 → ASR（Whisper / 字节 ASR）转口播 → OCR 提字幕 → 把"文字信息"喂文本大模型整合成步骤。
- ✅ 更可控、可缓存、**成本低**（文本 token 便宜）、长视频友好；口播是做菜视频信息量最大的部分，ASR 往往就够。
- ❌ 工程量大；抽帧策略难调；纯文本会丢失"没解说的手法"这类纯视觉信息。

**对比小结**

| 维度 | A 多模态直吃 | B 自建管线 |
| --- | --- | --- |
| 出效果速度 | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| 单次成本 | 中~高（随时长涨） | 低 |
| 长视频友好 | 差 | 好 |
| 纯视觉手法捕捉 | 好 | 差（需补抽帧） |
| 工程量 | 小 | 大 |
| 可控/可调试 | 弱 | 强 |

**我的推荐（分阶段）**：
- **MVP：走方案 A**，用豆包视频理解或 Gemini 直接出菜谱，最快验证"解析+讲解"的核心价值。个人自用量小，成本完全可接受（单个视频大致几毛~几块）。
- **规模化：转混合方案**——先用 ASR 拿口播 + OCR 拿字幕出草稿（便宜、覆盖 80%），只对"信息不足的步骤"稀疏抽帧补多模态调用。既降本又保质。

### 5.6 「为什么」讲解生成（独立一步）
- 拿到结构化步骤后，**单独**用一个强推理文本模型（带烹饪知识，如 DeepSeek / 豆包 / GPT 系）对每步生成 `why` 三段式。
- 与解析解耦的好处：可用更擅长"讲原理"的模型，且能对同一步反复"追问"而不重跑视频解析。
- 降幻觉：MVP 靠 prompt 约束 + `confidence` 兜底；V2 挂**烹饪原理知识库 RAG**（美拉德、乳化、焯水去腥等常见原理库）。

### 5.7 存储
- 结构化菜谱 / 收藏：Postgres（或 MVP 期 SQLite）。
- 视频文件 / 抽帧图：对象存储（S3 兼容 / 本地）。
- **解析结果缓存**：同一视频 URL 命中直接返回，省钱省时。

### 5.8 成本量级（个人自用）
- 单视频：方案 A 视频理解 几毛~几块 + 讲解文本 几分钱；缓存后重复访问≈0。
- 个人使用每月几十~上百次，月成本大概率在几十元内，完全可承受。

## 6. 分阶段路线图

| 阶段 | 目标 | 关键交付 |
| --- | --- | --- |
| **MVP（先验证核心价值）** | 证明"解析+讲原理"真的有用、够准 | 本地上传+YT/B站；方案A解析；三段式「为什么」；单端跟做模式（一步一屏+常亮+下一步）；收藏菜/收藏步骤；本地存储 |
| **V1** | 变顺手、可日常用 | 抖音/小红书源；语音翻步；步骤计时器；时间戳跳转；食材替代/份量换算；解析历史；双端补齐 |
| **V2** | 建壁垒 | RAG 原理知识库；术语微科普；追问对话；多视频对比；账号云同步；混合降本管线 |

> **给你的落地建议**：核心风险不在"App 壳"，而在**"AI 讲的原理准不准、有没有用"**。所以 MVP 别急着雕双端 UI，先用最小界面把"一个视频 → 一份带深度讲解的步骤表"跑通，你自己拿 5~10 个真实视频试，确认讲解质量过关，再投入做双端和跟做体验。

## 7. 主要风险 & 注意点
1. **版权/合规**：抓取抖音/小红书视频有灰区。个人自用风险低；若走向产品化，需转向"用户上传 / 官方授权 / 只存解析结果不存原视频"。
2. **AI 幻觉**：讲"为什么"最容易一本正经胡说。必须有 `confidence` 兜底 + 明确标注推测，宁可说"原视频没讲清"也别编。
3. **抖音/小红书解析易失效**：平台改版会打断取流，需持续维护，或引导用户改用"录屏上传"。
4. **解析耗时**：几十秒~几分钟，必须做异步 + 进度反馈，否则体验崩。
5. **成本随时长非线性上涨**（方案A）：对长视频要做时长上限或转混合方案。

## 9. v0.2 已落地：解析+讲解引擎（`tools/paoding/`）

已按"先做引擎、后做 App 壳"的策略，实现 MVP 第一块地基——一个零 npm 依赖的 Node CLI，跑通完整管线并端到端验证（除需 API key 的 LLM/ASR 调用外）。

**已确认的 MVP 决策**：先做引擎（非 App）｜中文+中式家常菜｜讲解深度可选(默认 balanced)｜本地文件 + yt-dlp 支持源｜无账号、输出本地 JSON+MD｜LLM 走 OpenAI 兼容接口｜每步带 `confidence` 防幻觉。

**管线**：`视频 →[yt-dlp]下载+抓简介 →[ffmpeg]抽音轨 →[ASR]口播转文字 →[LLM]整理结构化菜谱 →[LLM]逐步生成「为什么」→ JSON + Markdown`

**代码结构**：
- `bin/paoding.mjs` — CLI 入口（`--depth/--out/--keep-transcript`）
- `src/config.mjs` — .env/环境变量配置
- `src/download.mjs` — yt-dlp 下载 + 元数据 + ffmpeg 抽音轨
- `src/transcribe.mjs` — Whisper 兼容 ASR
- `src/chef.mjs` — 转写→结构化菜谱（第一步 LLM）
- `src/explain.mjs` — **逐步「为什么」讲解（核心差异，第二步 LLM）**
- `src/render.mjs` — JSON→可读 Markdown（雷区/低置信度标注）
- `src/pipeline.mjs` — 编排

产出的 `<菜名>.json`（每步含 `why{reason,if_not,cue}`、`risk_level`、`confidence`）即为**未来 App 的数据契约**。

**下一步（待引擎质量验证通过后）**：① 用真实视频调 chef/explain 两个 prompt 到满意；② 补抽帧+视觉 OCR 兜口播不足；③ 包 React Native App（跟做模式 + 收藏）。
```
