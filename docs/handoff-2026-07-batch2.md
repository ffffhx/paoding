## B1：任务持久化 + 排队（替代 429）

**现状**：解析任务存内存 `Map`（`app/server.mjs` 的 jobs），进程重启全丢——而 deploy workflow 每次 push 都重启服务，正在跑的解析白费；`MAX_RUNNING=2` 满了直接 429，无排队。

**要做**：
1. 任务元数据（id、类型、参数、状态、进度、结果菜谱 id、错误信息、时间戳）落盘到一个 `jobs/` 目录（每任务一个 JSON 文件，与 recipes 同级），状态变更时写入。启动时扫描：`running` 状态的任务标记为 `interrupted`（附提示"服务重启，任务中断，请重新发起"），供前端查询展示；不要求自动续跑（解析中间态难恢复，明确不做）。
2. 槽位满时任务进入 FIFO 内存队列（队列元数据也落盘），SSE 进度里报"排队中，第 N 位"；槽位释放时按序启动。队列上限（如 10，环境变量可调）超出才 429。
3. 新增 `GET /api/jobs`（最近 N 条任务及状态），前端首页可选展示"最近任务"入口（轻量即可）。
4. 完成态任务文件保留最近 50 条，超出按时间清理。
5. 测试：队列纯逻辑（入队/出队/上限）+ 重启后 interrupted 标记（server 测试里可模拟写一个 running 任务文件再起实例）。

## B2：网络请求重试退避

**现状**：LLM/ASR/网页抓取的 fetch 一次失败整趟报废，只有 JSON 解析层有一次重试。

**要做**：在 `src/` 抽一个 `fetchWithRetry`（或包装函数）：对网络错误与 5xx/429 响应做最多 2 次指数退避重试（如 1s/4s，可配），**必须尊重传入的 AbortSignal**（超时强杀链路不能被重试拖长：signal 已 abort 时立即放弃）。接入 `src/llm.mjs`、`src/transcribe.mjs`（云端路径）、`src/fetchText.mjs`。4xx（除 429）不重试。纯函数逻辑配测试（用本地 http 服务或注入 fetch stub）。

## B3：大文件上传流式落盘

**现状**：`readBody` 把上传整段 `Buffer.concat` 进内存（上限 800MB）再写临时文件，一个大上传吃几百 MB RAM。

**要做**：`POST /api/parse-file` 改为请求流直接 `pipe` 到临时文件（`fs.createWriteStream`），边收边写，累计超限即销毁请求并 413；其余 JSON 小体积端点维持现状。注意保留现有"先占槽再收 body"的并发竞态防护语义。测试：server 测试里上传一个中等 body 验证行为不回归。

## B4：userdata 并发写保护

**现状**：`PUT /api/userdata` 全量覆盖、last-write-wins，两台设备同时改互相覆盖。

**要做**：userdata JSON 加 `rev`（自增版本号）。PUT 时带上客户端读到的 `rev`：服务端一致则写入并 rev+1；不一致返回 409 + 服务端当前数据，前端收到 409 后做字段级合并（收藏/笔记/购物清单按并集或较新时间戳合并）再重试一次。前端 `store` 同步层相应改造。合并纯函数放可测试区并配测试。兼容旧数据（无 rev 视为 0）。

## B5：购物清单按超市货架分区

**现状**：购物清单已有同名合并 + 品类归类。竞品（ReciMe/Flavorish/Paprika）标配是按超市货架动线分区展示。

**要做**：
1. 扩充/固化品类映射为货架分区（如：蔬菜水果 / 肉禽蛋 / 水产 / 调味干货 / 粮油米面 / 乳品豆制品 / 冷冻 / 其他），映射表数据化（常见中文食材关键词→分区），LLM 不参与（离线可用）。
2. 前端购物清单按分区分组展示、分区内可勾选已购（已购沉底/划线）、支持"复制为文本"按分区输出。
3. 映射纯函数 + 分组逻辑配前端测试。

## B6：server.mjs AI 端点表驱动重构

**现状**：十几个 AI 端点（ask/nutrition/substitute/rescue/design 等）是高度重复的"读 JSON → loadRecipe → chatText → sendJSON"样板。

**要做**：抽成配置表（路径 → {校验、prompt 构造、LLM 调用方式}）+ 一个通用 handler，行为完全不变（响应格式、错误信息、鉴权/限流路径都不能变）。这是纯重构：改完 `npm test` 全绿 + 手动对拍一个端点的响应结构。若发现行为有歧义处，保守保持原样。
