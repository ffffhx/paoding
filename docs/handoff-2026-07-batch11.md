## K1：片尾/静态文字帧捕获（本批核心）

1. **抽帧策略修改**（src/vision.mjs）：
   - 场景切换抽帧保持现状，但**预留专门配额给片尾**：无论场景抽帧结果如何，额外在视频最后 20%（至少最后 30 秒，短视频按比例）均匀补抽 N 帧（如 6 帧，可调），与现有帧合并去重（按时间戳邻近去重）。
   - 片头 5 秒也补 1–2 帧（有些博主把配料表放开头）。
2. **配方卡识别与转录**：VL 批量读帧的 prompt 增强：遇到"整屏文字/列表排版/配料表/配方卡"类画面时，**逐字完整转录**（含用量数字与单位），并在输出中用『【画面配方卡】』标记该段。
3. **结构化优先级**：structureRecipe 的 user 内容里，画面配方卡文本单独成段并说明：「以下来自视频画面中的配方卡/配料表，用量以它为准（比口播记忆更可靠），与口播冲突时优先采用，并在对应食材 note 里注明 出处：画面配方卡」。
4. 测试：vision stub 返回带【画面配方卡】标记的转录 → 断言进入 structure 输入且菜谱用量来自卡片；抽帧函数的"片尾补抽时间点计算"抽成纯函数配测试（给定时长→返回补抽时间点列表）。

## K2：批量制备 vs 单份组装语义

1. structureRecipe prompt 新增规则：「饮品/酱料/卤水类视频常见『先批量制备基底（如一壶茶汤、一锅酱），再按杯/按份组装』。要识别这种结构：
   - servings 写清楚基准，如 茶汤约可做4杯；本菜谱按1杯组装。
   - 批量制备的食材在 note 标 批量：一次制备，约供N份；单份组装的食材标 单份用量。
   - 能从数量推算出份数时（如 1000ml 茶汤、每杯用 250ml → 约4杯）在 servings 或总览里写明推算，并标注是推算。」
2. 前端不需要新 UI：note 已有展示。份量缩放沿用现有 qty 机制（批量食材的缩放歧义在 note 里已说明基准，v1 不做双基准联动缩放，README 注明）。
3. 测试：LLM stub 输出带批量/单份 note 的菜谱 → 断言落库展示不丢。
