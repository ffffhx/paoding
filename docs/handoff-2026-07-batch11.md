## K1：片尾/静态文字帧捕获（本批核心）

1. **抽帧策略修改**（src/vision.mjs）：
   - 场景切换抽帧保持现状，但**预留专门配额给片尾**：无论场景抽帧结果如何，额外在视频最后 20%（至少最后 30 秒，短视频按比例）均匀补抽 N 帧（如 6 帧，可调），与现有帧合并去重（按时间戳邻近去重）。
   - 片头 5 秒也补 1–2 帧（有些博主把配料表放开头）。
2. **配方卡识别与转录**：VL 批量读帧的 prompt 增强：遇到"整屏文字/列表排版/配料表/配方卡"类画面时，**逐字完整转录**（含用量数字与单位），并在输出中用『【画面配方卡】』标记该段。
3. **结构化优先级**：structureRecipe 的 user 内容里，画面配方卡文本单独成段并说明：「以下来自视频画面中的配方卡/配料表，用量以它为准（比口播记忆更可靠），与口播冲突时优先采用，并在对应食材 note 里注明 出处：画面配方卡」。
4. 测试：vision stub 返回带【画面配方卡】标记的转录 → 断言进入 structure 输入且菜谱用量来自卡片；抽帧函数的"片尾补抽时间点计算"抽成纯函数配测试（给定时长→返回补抽时间点列表）。

## K2：批量制备 vs 单份组装 —— 结构化双板块（用户明确要求两个维度都完整展示）

用户原话：「批量的是一个菜谱，单份的是一个菜谱。或者在同一页上分成两栏，或者用 tab 标签页来切换。反正这两个维度都会展示，这两个我都要。」

**1. Schema（引擎层）**：
- ingredients 和 steps 增加可选字段 `phase: "batch" | "serving"`（普通菜谱不输出该字段，一切照旧——这是兼容安全网）。
- 菜谱顶层增加可选 `batch_info: { yield: "一壶茶汤（约1300毫升）", makes_servings: 4, makes_note: "按每杯250毫升推算（推算）", serving_desc: "以下为单杯用量" }`。
- structureRecipe prompt 规则：「饮品/酱料/卤水/高汤类视频常见『先批量制备基底，再按杯/按份组装』。识别到这种结构时：给每个食材和步骤标 phase（批量制备=batch，单份组装=serving）；输出 batch_info，能从数量推算份数时（1000ml÷每杯250ml≈4杯）写进 makes_servings 并在 makes_note 标注推算。识别不出明确结构时不要硬拆，不输出 phase。」
- 防御清洗：phase 只接受两个合法值，非法丢弃字段；只有部分条目带 phase 时视为无效结构（全体清掉 phase），避免半拆状态。

**2. 前端（两个维度都完整展示）**：
- 详情页：有 phase 时食材与步骤分成两个板块——「🫖 批量制备（基底）」和「🥤 单份组装」——同页上下两节 + 顶部锚点 tab 快速跳转（移动端友好，打印时两节自然顺排）。板块头显示 batch_info（"一壶约供4杯（推算）"）。
- **双缩放**：serving 板块沿用现有份量缩放（按杯数）；batch 板块独立一个「制备几批」倍数（默认 1，×2 就是两壶），板块头的"约供N杯"随批数联动重算。两个缩放互不干扰。
- 购物清单：batch 食材按批数、serving 食材按杯数分别换算后合并入清单。
- 跟做模式：批量步骤在前、单份步骤在后，两段之间插一个分隔屏（"基底完成 ✅ 下面按杯组装"）。
- 无 phase 的菜谱：渲染与现在完全一致（回归测试保障）。
- 新文案全部走 t()（zh+en）。

**3. 测试**：phase 清洗（半拆→清空）、双缩放换算纯函数、购物清单合并、无 phase 零回归；LLM stub 出带 phase 的菜谱端到端落库展示。
