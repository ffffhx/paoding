# 交接文档（第十七批）：管线性能优化 +抖音路径（2026-07）

> 真实验证数据：单视频全链路 434–958 秒，太慢。本批目标：**在不牺牲质量与诚实原则的前提下显著提速**，并补测抖音路径。
> 发现问题修复+测试+单独 commit。**绝对不要 git push**。额度红线 weekly 60%。

## Q1：耗时剖析仪表（先测量再优化）

管线各阶段（下载/抽音/ASR/视觉读屏/结构化/讲解/步骤图/食材图）耗时已有部分日志——统一落进菜谱 meta（`timings: {stage: seconds}`）+ CLI 结束时打印分解表。跑一条已验证过的短视频取基线数据，把分解结果写进本批报告。纯函数（计时聚合）配测试。

## Q2：按剖析结果优化 TOP 瓶颈（预期是 LLM/VL 侧）

候选手段（按剖析数据选做，优化后用同一条视频对比提速幅度）：
1. **讲解阶段**：explainSteps 若是逐步串行调用 LLM，改为受控并发（Ollama 会排队但可流水线化 prompt 准备）或按步骤分组批量（一次调用生成多步讲解，注意输出 JSON 结构稳定性——若质量下降就回退）。
2. **视觉读屏**：帧数自适应（短视频少抽帧）；跳过与前一帧感知哈希近似的重复帧（纯 Node 实现简易 dHash，零依赖）。
3. **步骤图/食材图**：已有并发与超时；检查 VL 挑帧是否存在重复请求同一帧、候选帧数量是否可自适应收缩。
4. **结构化**：prompt 里转写文本若整段重复传两次（结构化+讲解），评估讲解阶段是否可用精简上下文。
每项优化独立 commit，stub 测试保障行为不回归；报告里给出前后耗时对比（同一视频重跑）。

## Q3：抖音路径探测

yt-dlp 对抖音支持时好时坏：拿 1–2 个抖音视频链接（网页搜索"抖音 视频 链接 做菜"或用分享短链格式）测试 parse-url。跑通则全链路验证一条；跑不通则确认降级路径（自动转图文兜底？错误提示是否引导用户）如实记录。这决定 README 里对抖音的承诺要不要加限定语——按实测结果更新 README（双语）。
