## F1：拍照/截图导入菜谱（本批核心）

**场景**：用户拍菜谱书某页、手写菜谱、小红书截图（1–N 张图）→ VL 读出文字 → LLM 结构化成菜谱。

**要做**：
1. `src/` 新增图片管线入口（如 `processImages(paths, opts)`）：
   - 每张图走现有 VL 调用方式（参考 `vision.mjs` 的图片→模型请求封装，复用其配置/模型名），prompt 要求原样转录图中菜谱文字（标题/食材/步骤/用量），多张图按序拼接；
   - 转录文本走现有 `structureRecipe` + `explainSteps`（图片来源没有时间戳：`source_time` 留空，不硬造）；
   - 菜谱标 `source_type: "image"`，VL 读不出可用文字时明确报错（"图片中未识别到菜谱内容"），不臆造。
2. `POST /api/parse-images`：multipart 或多次上传复用现有上传通道（看现有 parse-file 的实现选最小改动方案），限制张数（默认 6）与单张大小，走鉴权+限流+任务队列（B1 的 job 体系，可后台+SSE 进度）。
3. 前端：解析入口加「拍照/图片导入」，移动端 `<input type="file" accept="image/*" capture multiple>`；进度与错误提示复用现有任务 UI。
4. 测试：用 E1 的 LLM/VL stub 方式端到端测 processImages（stub VL 返回固定转录文本 → 断言菜谱结构、source_time 为空、imported 语义正确；VL 返回空 → 报错路径）。
5. VL 未配置（无视觉模型）时的降级：接口返回明确错误提示"需配置视觉模型"，README 说明。

## F2：分享页互通

**现状**：`/r/<id>` 只读分享页可看菜谱；C3 有 JSON-LD 导入端点。两者没打通——别人看到分享页只能看。

**要做**：
1. 分享页加「导出 JSON-LD」按钮（页面内直接生成下载，无需鉴权——分享页本就公开该菜谱）；再加一段"复制到自己的庖丁"说明（下载 → 自己实例设置页导入）。
2. 分享页补营养卡片与技法标注展示（有才显示）。
3. 分享页保持无鉴权只读、不能触发任何写操作/LLM 调用（自查确认）。
4. 有 server 测试保障：分享页 200、包含 JSON-LD `<script type="application/ld+json">`（顺带让搜索引擎能索引菜谱——SEO 白赚）。

## F3：技法库 v2 —— LLM 跨菜归纳

**现状**：C4 的技法页是关键词命中聚合（哪些菜的哪些步骤用了焯水）。

**要做**：
1. `POST /api/techniques/:name/summary`（走鉴权+限流）：把该技法在所有菜谱里的步骤文本+why 摘录喂给 LLM，生成「技法通用要点」：什么时候用/关键判断/常见翻车点，输出结构化 JSON（三字段），**结果缓存落盘**（`techniques-cache/` 或并入现有数据目录），出现该技法的菜谱集合变化时失效。
2. 前端技法详情页加「AI 归纳要点」按钮（点击生成，带"AI 归纳，仅供参考"标注），已有缓存直接展示。
3. 测试：LLM stub 返回固定 JSON → 断言缓存写入与二次请求直读缓存；集合变化失效逻辑纯函数化配测试。
